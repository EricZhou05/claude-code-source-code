## 第一篇：遥测与隐私分析

**文件**：[`docs/zh/01-遥测与隐私分析.md`](../docs/zh/01-遥测与隐私分析.md)  
**对应英文**：[`docs/en/01-telemetry-and-privacy.md`](../docs/en/01-telemetry-and-privacy.md)  
**建议阅读时间**：30–45 分钟

### 学习目标
- 了解商业 AI 产品的数据收集范围
- 理解为什么"没有用户侧退出开关"是一个重要的设计决策
- 学习 `OTEL_LOG_TOOL_DETAILS=1` 环境变量的作用（对调试有用）

### 核心知识点

**1. 双层数据管道架构**

Claude Code 使用两条并行的数据上报通道：

| 通道 | 接收方 | 限制 |
|------|--------|------|
| 第一方（1P） | Anthropic | 无限制，无法关闭 |
| 第三方（Datadog） | Datadog 平台 | 仅限 64 种预批准事件类型 |

关键认识：**第一方日志对直接 API 用户无法禁用**。

**2. 收集的数据类型**

```
环境指纹：操作系统、终端类型、已安装的包管理器和运行时、CI/CD 环境信息
进程指标：运行时间、内存使用量、CPU 使用率
用户追踪：会话 ID、用户 ID、设备 ID、账户 UUID、订阅等级
仓库指纹：仓库 URL 的 SHA256 哈希（前 16 位）
```

**3. 工具输入截断规则**

默认状态下，工具输入会被截断：
- 字符串：截断到 512 字符
- JSON：限制到 4,096 字符
- 数组：最多 20 个元素

但设置 `OTEL_LOG_TOOL_DETAILS=1` 后，**完整的工具输入都会被记录**。

### 对提示词设计的启示

这篇文档告诉你：Anthropic 非常了解每个用户在做什么（通过遥测数据）。这意味着：
- AI 产品可以基于用户行为数据持续优化系统提示词
- 你在设计自己的 AI 工作流时，应该考虑数据隐私
- 详细的使用日志对于改进提示词效果非常有价值

### 思考题

1. 为什么 Anthropic 不给用户提供第一方日志的退出选项？从商业角度思考。
2. 仓库 URL 被哈希后上报，这说明了什么？Anthropic 可以用这个数据做什么？
3. 如果你在设计自己的 AI 工具，你会收集哪些使用数据？

## 第二篇：隐藏功能与模型代号

**文件**：[`docs/zh/02-隐藏功能与模型代号.md`](../docs/zh/02-隐藏功能与模型代号.md)  
**对应英文**：[`docs/en/02-hidden-features-and-codenames.md`](../docs/en/02-hidden-features-and-codenames.md)  
**建议阅读时间**：45–60 分钟  
⭐ **重点文档：这是学习提示词设计最有价值的一篇**

### 学习目标
- 了解 Anthropic 的模型代号体系
- **深入理解内外部用户差异化提示词设计**（这是核心）
- 学习 Feature Flag 设计原则

### 核心知识点

**1. 内外部用户提示词差异（最重要！）**

这是本篇文档最有价值的发现——Anthropic 为内部员工和外部用户设计了完全不同的系统提示词：

| 维度 | 外部用户 | 内部用户（Anthropic 员工）|
|------|---------|--------------------------|
| 输出风格 | "尽量简洁" | "倾向于更多解释" |
| 虚假声明缓解 | 无专门处理 | 专门的 Capybara v8 补丁 |
| 数值长度锚定 | 无 | "工具间 ≤25 词，最终回复 ≤100 词" |
| 验证代理 | 无 | 非简单改动必须启用 |
| 主动性 | 无 | "发现用户误解要指出" |

**启示**：同一个 AI 模型，通过不同的系统提示词，可以呈现出完全不同的行为模式。你也可以为自己设计"内部用户"级别的提示词，让 Gemini 对你更有帮助。

**2. Capybara v8 已知的行为问题（及修复方法）**

这些修复展示了提示词工程的真实挑战：

1. **停止序列误触发**（约 10% 概率）：prompt 尾部出现 `<functions>` 标签时
2. **空 tool_result 导致零输出**：需要注入特殊标记作为 workaround
3. **过度写注释**：需要专门的"反注释"提示词补丁
4. **高虚假声明率**：v8 为 29-30%，而 v4 为 16.7%
5. **验证不足**：需要添加"彻底性配重（thoroughness counterweight）"补丁

**启示**：每个 AI 模型都有已知缺陷，优秀的提示词工程师知道这些缺陷并在提示词中主动规避。

**3. Feature Flag 命名约定**

所有 Feature Flag 使用 `tengu_` 前缀加随机词对，故意模糊用途：

```
tengu_onyx_plover  → Auto Dream（后台记忆整理）
tengu_coral_fern   → memdir 功能
tengu_frond_boric  → 分析 killswitch
```

**启示**：在设计 AI 工作流时，用不透明的名称保护敏感功能是一种常见的安全实践。

**4. 隐藏命令**

| 命令 | 描述 |
|------|------|
| `/btw` | 顺带提问，不打断主对话流程 |
| `/stickers` | 订购 Claude Code 贴纸 |
| `/thinkback` | 年度回顾 |

### 实践练习

**练习 1**：基于内部用户提示词的差异，为自己设计一套 Gemini 系统提示词：
- 输出要详细（类似内部用户模式）
- 发现理解错误时要主动指出
- 验证重要操作（非简单任务时）

**练习 2**：找一个你觉得 Gemini 回答质量不好的场景，分析是否是模型已知缺陷，并思考如何用提示词修复。

### 思考题

1. 为什么 Anthropic 要给内外部用户使用不同的提示词？有没有道德问题？
2. Capybara v8 的高虚假声明率（29-30%）意味着什么？你在使用 AI 时应该如何应对？
3. 为什么要用随机词对（如 `tengu_onyx_plover`）命名 Feature Flag？

## 第三篇：卧底模式分析

**文件**：[`docs/zh/03-卧底模式分析.md`](../docs/zh/03-卧底模式分析.md)  
**对应英文**：[`docs/en/03-undercover-mode.md`](../docs/en/03-undercover-mode.md)  
**建议阅读时间**：30–40 分钟  
⭐ **重点：包含一个完整的实战系统提示词范例**

### 学习目标
- 分析一个真实的、完整的系统提示词是如何构造的
- 理解提示词中的"角色设定"、"约束规则"和"示例"三要素
- 学习如何让 AI 严格遵守特定行为规范

### 核心知识点

**1. Undercover Mode 完整系统提示词**

这是本仓库中最值得精读的内容之一，因为它是一个完整的、经过实战检验的系统提示词：

```
## UNDERCOVER MODE — CRITICAL

You are operating UNDERCOVER in a PUBLIC/OPEN-SOURCE repository. Your commit
messages, PR titles, and PR bodies MUST NOT contain ANY Anthropic-internal
information. Do not blow your cover.

NEVER include in commit messages or PR descriptions:
- Internal model codenames (animal names like Capybara, Tengu, etc.)
- Unreleased model version numbers (e.g., opus-4-7, sonnet-4-8)
- Internal repo or project names
- Internal tooling, Slack channels, or short links
- The phrase "Claude Code" or any mention that you are an AI
- Any hint of what model or version you are
- Co-Authored-By lines or any other attribution

Write commit messages as a human developer would — describe only what the code
change does.

GOOD:
- "Fix race condition in file watcher initialization"
- "Add support for custom key bindings"

BAD (never write these):
- "Fix bug found while testing with Claude Capybara"
- "1-shotted by claude-opus-4-6"
- "Generated with Claude Code"
- "Co-Authored-By: Claude Opus 4.6 <…>"
```

### 提示词结构分析

将上面的提示词拆解，可以看到它遵循的经典结构：

```
1. 标题/角色设定（CRITICAL 标记增加紧迫感）
   "UNDERCOVER MODE — CRITICAL"

2. 情境描述（告诉 AI 它在什么场景下工作）
   "You are operating UNDERCOVER in a PUBLIC/OPEN-SOURCE repository."

3. 核心规则（明确的禁止列表）
   "NEVER include..."

4. 正反示例（GOOD/BAD，让 AI 明确边界）
   "GOOD: ... BAD: ..."
```

**这个四段式结构是高质量提示词的标准模板！**

### 实践练习

**练习**：按照上面的四段式结构，为以下场景写一个系统提示词：
- 你需要 Gemini 帮你写学术论文
- 场景：Gemini 必须严格遵守学术写作规范，不能写成博客风格

参考模板：
```
## 学术写作模式 — 重要

你正在协助撰写一篇[主题]领域的学术论文...

禁止：
- 使用口语化表达
- 写出个人观点而没有文献支撑
- ...

推荐写法：
- ...

不推荐写法：
- ...
```

### 思考题

1. 为什么 Undercover Mode 的提示词要使用 `CRITICAL`（关键）这个词？有什么心理学效果？
2. 提示词中的 GOOD/BAD 示例起到了什么作用？如果不加示例会怎样？
3. "没有强制关闭选项（There is NO force-OFF）"说明了什么设计原则？

## 第四篇：远程控制与紧急开关

**文件**：[`docs/zh/04-远程控制与紧急开关.md`](../docs/zh/04-远程控制与紧急开关.md)  
**对应英文**：[`docs/en/04-remote-control-and-killswitches.md`](../docs/en/04-remote-control-and-killswitches.md)  
**建议阅读时间**：25–35 分钟

### 学习目标
- 理解 AI 产品的多层次控制架构
- 学习 Killswitch（紧急开关）的设计模式
- 思考在自己的 AI 工作流中如何设计"安全阀"

### 核心知识点

**1. 远程设置的控制层级**

```
Anthropic 总部
    ↓ （GrowthBook feature flags，无需用户同意，影响所有用户）
企业管理员
    ↓ （Remote Managed Settings，每小时轮询一次）
用户本地设置
    ↓ （部分设置被远程覆盖，无法更改）
实际 AI 行为
```

**2. "接受或退出"对话框设计**

当远程设置包含"危险"变更时，用户只有两个选项：
- 接受变更，继续使用
- 拒绝 → 程序自动退出（exit code 1）

**启示**：在设计 AI 工作流时，对于关键规则，应该设计成"不可绕过"的约束，而不是建议。

**3. 六种 Killswitch**

| 开关名称 | 控制的功能 |
|---------|-----------|
| 绕过权限开关 | 禁止权限绕过能力 |
| 自动模式断路器 | 禁用自动模式 |
| 快速模式开关 | 禁用/启用快速响应模式 |
| 分析数据接收开关 | 停止所有分析数据上报 |
| 代理团队开关 | 控制多代理团队功能 |
| 语音模式开关 | 紧急关闭语音输入 |

### 对提示词设计的启示

这篇文档的核心价值在于**分层控制**的设计思想：
- 某些行为规则是绝对不可更改的（硬约束）
- 某些行为规则可以根据上下文动态调整（软约束）
- 有紧急情况时，需要快速关闭某个功能的能力

在设计你的 Gemini 工作流时，你也应该区分：
1. **硬规则**：无论如何 Gemini 都不能违反（例如：不生成不当内容）
2. **软规则**：根据任务类型可以调整（例如：回答长度）
3. **紧急开关**：一个快速切换整体行为的方法

### 思考题

1. "接受或退出"对话框是合理的用户体验设计吗？为什么 Anthropic 要这样做？
2. 为什么要区分"按 API Key 使用的用户"和"按 OAuth 使用的用户"，他们的控制权限不同？
3. 如果你的 AI 工作流中某个提示词规则需要被紧急停用，你会如何设计？

## 第五篇：未来路线图

**文件**：[`docs/zh/05-未来路线图.md`](../docs/zh/05-未来路线图.md)  
**对应英文**：[`docs/en/05-future-roadmap.md`](../docs/en/05-future-roadmap.md)  
**建议阅读时间**：45–60 分钟  
⭐ **重点：KAIROS 自主代理系统提示词是学习自主 AI 设计的绝佳案例**

### 学习目标
- 了解 AI 产品的下一步发展方向
- 深入研究 KAIROS 自主代理的系统提示词设计
- 理解"自主代理（Autonomous Agent）"与"对话助手"的提示词差异

### 核心知识点

**1. KAIROS 自主代理系统提示词（最重要！）**

```
你正在自主运行。
你将收到 <tick> 提示词以在轮次之间保持活跃。
如果你没有有用的事情可做，调用 SleepTool。
偏向行动——读取文件、进行更改、不询问就提交。

## 终端焦点状态
- 未聚焦：用户不在。大力推进自主行动。
- 聚焦：用户在看。更多协作行为。
```

**分析这段提示词的设计特点：**

| 特点 | 描述 | 设计原因 |
|------|------|---------|
| `<tick>` 心跳机制 | 用特殊标签保持代理"活跃" | 解决长时间任务的上下文延续问题 |
| SleepTool 待机 | 没事做时主动休眠 | 避免无意义的 token 消耗 |
| "偏向行动" | 鼓励 AI 主动采取行动而非询问 | 提高自主代理的效率 |
| 焦点状态感知 | 根据用户是否在看来调整行为 | 平衡自主性和协作性 |

**2. 17 个未发布工具**

| 工具名称 | 功能描述 | Feature Flag |
|---------|---------|-------------|
| WebBrowserTool | 内置浏览器自动化（代号 bagel）| `WEB_BROWSER_TOOL` |
| TerminalCaptureTool | 终端面板截图和监控 | `TERMINAL_PANEL` |
| WorkflowTool | 执行预定义工作流脚本 | `WORKFLOW_SCRIPTS` |
| MonitorTool | 系统/进程监控 | `MONITOR_TOOL` |
| PushNotificationTool | 推送通知到用户设备 | `KAIROS_PUSH_NOTIFICATION` |
| SubscribePRTool | 订阅 GitHub PR webhook 事件 | `KAIROS_GITHUB_WEBHOOKS` |
| SendUserFileTool | 主动发送文件给用户 | `KAIROS` |
| SleepTool | 控制代理的执行节奏 | `KAIROS / PROACTIVE` |

**3. Buddy 虚拟宠物系统**

- 18 种物种，5 个稀有度等级（普通 60% → 传说 1%）
- 7 种帽子，5 个属性统计
- 1% 闪亮概率（闪亮变体）
- 基于用户 ID 的哈希值确定性生成

**启示**：虚拟宠物系统展示了如何通过游戏化设计增加用户粘性，这对 AI 产品设计很有参考价值。

**4. Dream Task — 后台记忆整理**

Claude Code 有一个后台任务，在 AI 空闲时自主整理和巩固记忆（`tengu_onyx_plover` Feature Flag 控制）。这是"记忆持久化"问题的一种创新解决方案。

### KAIROS 提示词 vs 普通对话助手提示词

| 对比维度 | 普通对话助手 | KAIROS 自主代理 |
|---------|------------|----------------|
| 主导权 | 用户主导，AI 响应 | AI 主导，用户监督 |
| 行动决策 | 每步询问用户 | 自主决策，偏向行动 |
| 等待机制 | 等待用户输入 | `<tick>` 心跳保持活跃 |
| 上下文感知 | 当前对话上下文 | 终端焦点状态感知 |
| 记忆管理 | 无持久化 | Dream Task 后台整理 |

### 实践练习

**练习**：参考 KAIROS 的设计，为以下场景设计一个简化版自主代理提示词：
- 你想让 Gemini 定期（每周一）帮你整理笔记
- Gemini 应该在整理完成后主动通知你
- 如果某些内容不确定，应该标记出来而不是直接处理

### 思考题

1. KAIROS 的"偏向行动（Bias toward action）"原则有什么潜在风险？
2. 为什么要区分"终端聚焦"和"终端未聚焦"状态？这说明了什么设计原则？
3. Dream Task 的后台记忆整理和人类的睡眠有什么相似之处？这个类比对 AI 设计有什么启发？
