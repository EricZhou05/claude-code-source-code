# 学习指南 05 — 上下文压缩与长对话管理 (Context Compression)

当 AI 对话变得非常长时，会面临两个问题：一是超出模型的上下文窗口限制（Context Window），二是 Token 消耗过快导致成本飙升。Claude Code 通过一套复杂的“压缩与总结”机制，在不丢失关键信息的前提下，实现了“无限长度”的对话体验。

本章节将分析 `src/services/compact/` 目录下的核心实现。

---

## 1. 自动触发机制 (Auto-Compact)

Claude Code 不会等到上下文完全填满才行动，而是设定了一个“安全水位线”。

### 源码原文 (`src/services/compact/autoCompact.ts`)
> ```typescript
> export function getAutoCompactThreshold(model: string): number {
>   const effectiveContextWindow = getEffectiveContextWindowSize(model)
>   const autocompactThreshold = effectiveContextWindow - AUTOCOMPACT_BUFFER_TOKENS
>   return autocompactThreshold
> }
> ```

### 技术解析
*   **缓冲带 (Buffer)**：代码中定义了 `AUTOCOMPACT_BUFFER_TOKENS` (约 13,000 tokens)。
*   **提前预警**：当当前 Token 数达到 `窗口大小 - 13K` 时，系统会自动触发压缩任务。这保证了在压缩过程中，模型仍有足够的空间输出总结内容。

---

## 2. 压缩的核心策略：两阶段总结

压缩不是简单地删掉旧消息，而是启动一个“微型代理（Forked Agent）”来生成高度浓缩的摘要。

### 源码原文 (`src/services/compact/prompt.ts`)
> ```typescript
> const BASE_COMPACT_PROMPT = `Your task is to create a detailed summary of the conversation so far... 
> 1. Primary Request and Intent: ...
> 3. Files and Code Sections: ...
> 4. Errors and fixes: ...
> 8. Current Work: ...
> 9. Optional Next Step: ...`
> ```

### 中文翻译
你的任务是为到目前为止的对话创建一个详细的总结……
1. 主要请求与意图：详细捕捉用户的所有明确请求。
3. 文件与代码段：枚举查看、修改或创建的具体文件及代码段。
4. 错误与修复：列出遇到的所有错误及解决方法。
8. 当前工作：详细描述总结请求前正在进行的具体工作。
9. 可选下一步：列出与当前工作直接相关的后续步骤。

### 教学解析
*   **结构化总结**：通过 9 个维度的硬性要求，确保总结内容具备“工程可执行性”，而不仅仅是文字摘要。
*   **<analysis> 思考垫片**：模型被要求先在 `<analysis>` 标签中进行思考，再输出正式总结。最终系统会通过 `formatCompactSummary` 函数剔除思考过程，只保留干货。

---

## 3. 关键信息的“防丢失”恢复 (Post-Compact Restore)

压缩后，模型会丢失对之前读取的文件内容的直接感知。为了弥补这一点，Claude Code 会自动“找回”最近的文件状态。

### 源码原文 (`src/services/compact/compact.ts`)
> ```typescript
> export async function createPostCompactFileAttachments(
>   readFileState: Record<string, { content: string; timestamp: number }>,
>   // ...
> ): Promise<AttachmentMessage[]> {
>   const recentFiles = Object.entries(readFileState)
>     .sort((a, b) => b.timestamp - a.timestamp)
>     .slice(0, maxFiles) // 默认恢复最近的 5 个文件
>   // ...
> }
> ```

### 技术解析
*   **最近最少使用 (LRU) 逻辑**：系统会记录所有被 `FileReadTool` 读取过的文件。压缩结束后，它会自动选取最近使用的 5 个文件，以附件形式（Attachment）重新注入到新上下文的开头。
*   **无缝衔接**：这使得模型在压缩后依然能“记住”正在编辑的文件内容，避免了重复读取的操作。

## 5. 手动干预：/compact 命令的精细控制

用户可以通过 `/compact` 命令手动触发压缩。与自动触发不同，手动压缩允许用户提供“自定义指令”来引导总结的方向。

### 源码原文 (`src/commands/compact/compact.ts`)
> ```typescript
> export const call: LocalCommandCall = async (args, context) => {
>   const customInstructions = args.trim()
>   // ...
>   // 在正式总结前，先运行微压缩 (Micro-compact)
>   const microcompactResult = await microcompactMessages(messages, context)
>   const messagesForCompact = microcompactResult.messages
>
>   const result = await compactConversation(
>     messagesForCompact,
>     context,
>     // ...
>     customInstructions,
>     false,
>   )
> }
> ```

### 技术解析
*   **微压缩 (Micro-compact) 预处理**：在进行昂贵的 LLM 总结之前，系统会先调用 `microcompactMessages`。这通常涉及一些基于规则的“硬删除”或“简单替换”（如删除冗长的 tool_use 原始输出，只保留关键状态），从而在不调用模型的情况下先节省一部分 Token。
*   **带意图的压缩**：如果用户输入 `/compact focus on the authentication logic`，`customInstructions` 会被传递给压缩 Prompt。这使得生成的摘要能根据用户的当前关注点进行“有损但高效”的过滤。
*   **缓存共享参数**：通过 `getCacheSharingParams`，手动压缩尝试与主对话共享系统提示词和工具定义，以最大化利用 Prompt Cache，减少压缩任务本身的冷启动延迟。

## 6. 响应式压缩 (Reactive Compact) 的兜底

除了主动（Proactive）和手动（Manual）压缩，Claude Code 还具备响应式（Reactive）能力。

### 源码分析
当 API 报出 `prompt_too_long` 错误时，`reactiveCompact` 会介入。它会采取更激进的策略，比如直接从消息列表头部（Head）进行物理切除，以确保会话不会因为 Token 溢出而彻底锁死。

## 迁移到 Gemini 工作流的实战建议 (续)

4.  **分级压缩策略**：
    *   **一级（微压缩）**：写一个简单的正则脚本，删除对话历史中冗长的日志输出或 Base64 图像数据。
    *   **二级（智能总结）**：调用 Gemini Pro 进行结构化总结。
    *   **三级（响应式切除）**：如果依然超限，直接丢弃最早的 20% 对话内容。
